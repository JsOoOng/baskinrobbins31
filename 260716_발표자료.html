발주 요청 승인/처리

1. 기능 개요

본사 관리자가 지점에서 요청한 발주를 조회하고 승인, 배송 처리, 완료 처리하는 기능이다.

지점은 필요한 재고 품목을 본사에 발주 요청하고, 본사는 해당 요청을 순차적으로 처리한다.

예를 들어 지점은 다음과 같은 품목을 발주할 수 있다.

파인트 컵 3 BOX
핑크 스푼 5 BOX
초코시럽 10 EA
드라이아이스 2 BOX

발주 요청은 처리 단계에 따라 상태가 변경된다.

WAITING → APPROVED → SHIPPING → COMPLETED

2. 담당자

담당자 담당 내용
천대광 발주 요청 목록 조회, 상세 조회, 승인, 배송 처리, 완료 처리 API 구현

3. 사용자

사용자 접근 가능 여부 설명
최고 관리자 가능 전체 발주 요청 처리 가능
본사 관리자 가능 발주 승인, 배송, 완료 처리 가능
지점 관리자 등록/조회 가능 본인 지점 발주 요청 및 상태 확인
고객 불가 내부 운영 기능이므로 접근 불가

4. 사용 테이블

RESTOCK_REQUESTS

컬럼 사용 목적

store_id 발주 요청 지점

status 발주 상태
admin_id 처리 본사 관리자

컬럼 사용 목적

APPROVED 승인 완료

배너 관리 CRUD

1. 기능 개요

본사 관리자가 키오스크 또는 고객 화면에 노출할 배너를 등록, 조회, 수정, 비활성화하는 기능이다.

배너는 이벤트, 공지사항, 프로모션 등을 고객에게 노출하는 데 사용된다.

예를 들어 다음과 같은 배너를 등록할 수 있다.

여름 시즌 할인 이벤트
아이스크림 케이크 예약 안내
멤버십 포인트 적립 안내

배너는 isActive 상태에 따라 노출 여부를 관리한다.

isActive = true  → 화면 노출 가능
isActive = false → 화면 노출 제외

본사 관리자 권한/보안 강화

1. 기능 개요

본사 관리자 기능을 JWT 인증 기반으로 보호하고, 최고 관리자와 일반 본사 관리자의 권한을 분리하는 기능이다.

기존에는 Postman 테스트 편의를 위해 requesterAdminId를 직접 전달하는 방식이었지만, 최종 구현에서는 로그인 후 발급받은 JWT 토큰을 이용하여 현재 로그인한 관리자를 식별한다.

기존 방식:
requesterAdminId 직접 전달

변경 방식:
JWT 로그인 → Authorization Bearer 토큰 → SecurityContext 인증 정보 사용

권한은 다음과 같이 구분한다.

SUPER_ADMIN → 관리자 생성, 권한 변경, 상태 변경 가능
ADMIN       → 일반 본사 기능 사용 가능, 관리자 관리 제한

2. 담당자

담당자 담당 내용
천대광 본사 관리자 JWT 로그인, 권한 분리, 관리자 보안 API 구현

3. 사용자

사용자 접근 가능 여부 설명
최고 관리자 가능 관리자 생성, 권한 변경, 상태 변경, 비밀번호 변경 가능
본사 관리자 일부 가능 본인 비밀번호 변경 가능, 관리자 관리 기능 제한
지점 관리자 불가 본사 관리자 보안 기능 접근 불가
고객 불가 내부 관리자 보안 기능이므로 접근 불가

바닐라

민트초코
엄마는 외계인
체리쥬빌레
뉴욕치즈케이크

isActive = true  → 운영 가능
isActive = false → 비활성화, 신규 지점 배정 불가

2. 발주 요청 승인/처리 DTO 개요

발주 요청 승인/처리 DTO는 지점에서 본사로 요청한 재고 발주를 본사 관리자가 조회하고, 승인, 배송 처리, 완료 처리할 때 사용한다.

지점은 운영에 필요한 재고 품목을 본사에 요청할 수 있다.

예시 발주 품목:

파인트 컵
쿼터 컵
스푼
냅킨
드라이아이스
초코시럽
포장 박스

발주 요청은 처리 단계에 따라 상태가 변경된다.

WAITING → APPROVED → SHIPPING → COMPLETED

🔴 중요 체크

발주 요청은 상태 흐름이 중요하다.

아무 상태에서나 바로 완료 처리하면 안 되고, 정해진 순서대로만 변경되어야 한다.

가능한 흐름
WAITING → APPROVED
APPROVED → SHIPPING
SHIPPING → COMPLETED

불가능한 흐름
WAITING → SHIPPING
WAITING → COMPLETED
APPROVED → COMPLETED
COMPLETED → APPROVED

현재 방식
→ Body.adminId 사용

→ JWT token에서 adminId 추출

최종적으로는 adminId를 RequestDTO에서 제거하고, SecurityContext에서 현재 로그인한 관리자를 가져오는 구조로 바꿀 수 있다.

2. 배너 관리 DTO 개요

배너 관리 DTO는 본사 관리자가 키오스크 또는 고객 화면에 노출할 배너를 등록, 수정, 조회, 비활성화할 때 사용한다.

배너는 이벤트, 공지사항, 신메뉴 홍보, 프로모션 안내 등을 고객 화면에 보여주기 위한 데이터다.

예시 배너:

7월 신메뉴 출시
여름 시즌 할인 이벤트
아이스크림 케이크 예약 안내
멤버십 포인트 적립 안내

배너는 isActive 값을 통해 노출 여부를 관리한다.

isActive = true  → 활성 상태, 고객 화면 노출 가능
isActive = false → 비활성 상태, 고객 화면 노출 제외

🔴 중요 체크

배너 삭제는 실제 DB 데이터를 삭제하는 방식이 아니라,

isActive = false로 변경하는 비활성화 방식이 더 안전하다.

DELETE /head/banners/{bannerId}
↓
isActive = false

이렇게 하면 과거에 사용했던 배너 기록을 DB에 남겨둘 수 있다.

2. 본사 매출/운영 통계 DTO 개요

본사 매출/운영 통계 DTO는 본사 관리자가 전체 매출, 지점별 매출, 상품별 판매량, 일자별 매출, 지출 통계를 조회할 때 사용한다.

통계 기능은 등록/수정/삭제 기능이 아니라, 이미 쌓여 있는 주문, 결제, 상품, 지점, 지출 데이터를 조회해서 집계 결과를 반환하는 기능이다.

예시 통계 항목:

총 결제 금액
쿠폰 할인 금액
포인트 사용 금액
총 지출 금액
순이익
지점별 매출
상품별 판매량
일자별 매출
지출 카테고리별 합계

🔴 중요 체크

통계 DTO는 대부분 RequestDTO가 없다.

이유는 현재 구현한 통계 API가 전부 GET 방식이고, Body로 값을 보내지 않기 때문이다.

GET /head/statistics/sales

GET /head/statistics/products
GET /head/statistics/period
GET /head/statistics/expenses

즉, 본사 매출/운영 통계 DTO는 전부 ResponseDTO 중심이다.

필드명 의미

5-5. 예시 의미

orderCount = 10
totalOrderAmount = 200000
totalPaidAmount = 180000
totalCouponDiscount = 15000
totalPointUsed = 5000
totalExpenseAmount = 50000

의미:

전체 주문은 10건이다.
주문 총액은 200000원이다.
실제 결제된 금액은 180000원이다.
쿠폰 할인은 15000원, 포인트 사용은 5000원이다.

5-7. 중요한 설계 기준

🔴 netProfit은 DB에 저장된 값이 아니라 계산된 값이다.

netProfit = totalPaidAmount - totalExpenseAmount

즉, netProfit은 별도 테이블 컬럼에서 가져오는 값이 아니라 Service에서 계산해서 DTO에 담는 값이다.

🔴 결제 금액은 payment_status = PAID 기준으로 계산한다.

매출 통계는 실제 결제가 완료된 데이터 중심으로 계산해야 한다.

결제 완료
→ PAID

통계 포함
→ payment_status = PAID

🔴 취소 주문은 통계에서 제외한다.

order_status = CANCELED
→ 통계 제외

취소된 주문까지 매출에 포함하면 실제 매출보다 크게 잡힐 수 있다.

목록 조회

→ 문의 내용, 답변 내용, 답변 작성 관리자까지 포함

→ 체리쥬빌레 운영 중, container = 1, isSoldOut = false

ICECREAM_FLAVORS는 본사 기준의 전체 맛 데이터이고,

    2. 내용

    지점 상품의 품절 상태를 변경하는 기능을 구현하면서 StoreProduct 기능을 복사하여 StoreFlavor 기능을 새로 작성하였다.

    DTO를 StoreFlavorStatusResponse로 변경하였지만, Service 메서드의 반환 타입은 기존 StoreProductStatusResponse로 남아 있었다.

    그 결과 StoreFlavorStatusResponse 객체를 반환하는 코드와 메서드의 반환 타입이 일치하지 않아 컴파일 오류가 발생하였다.

    Type mismatch: cannot convert from StoreFlavorStatusResponse to StoreProductStatusResponse

    3. 해결방안

    Service 메서드의 반환 타입을 실제 반환하는 DTO와 동일하게 수정하였다.

    수정 전

    java public StoreProductStatusResponse updateSoldOut(...) { return StoreFlavorStatusResponse.from(storeFlavor); }

    수정 후

    java public StoreFlavorStatusResponse updateSoldOut(...) { return StoreFlavorStatusResponse.from(storeFlavor); }

    또한 from() 메서드 내부에서도 StoreProduct가 아닌 StoreFlavor 엔티티를 사용하도록 수정하였다.

    반환 타입과 실제 반환 객체의 타입을 일치시켜 컴파일 오류가 해결되었다.

    💡

    내용 :

    프로젝트 실행 중 Spring Boot가 SecurityConfig Bean을 등록하는 과정에서 동일한 Bean 이름이 존재하여 애플리케이션 실행에 실패하였다.

    com.kiosk.common.config.SecurityConfig와 com.kiosk.branch.security.SecurityConfig 두 클래스가 모두 @Configuration으로 등록되어 있었으며, Spring은 기본적으로 클래스명을 camelCase 형태의 Bean 이름으로 등록하기 때문에 두 클래스 모두 securityConfig라는 동일한 이름을 가지게 되었다.

    이로 인해 Spring이 어떤 Bean을 등록해야 하는지 판단하지 못해 ConflictingBeanDefinitionException이 발생하였다.

    해결 방안 :

    프로젝트에서 사용할 SecurityConfig를 하나만 유지하도록 수정하고, 중복되는 SecurityConfig를 제거하였다.

    결과 :

    Bean 이름 충돌이 해결되어 Spring Boot가 정상적으로 Configuration 클래스를 등록할 수 있게 되었다.

  • 26.07.09 Order Bean 이름 충돌로 애플리케이션 실행 실패

    💡

    내용 :

    고객용 주문 기능과 지점용 주문 기능을 각각 구현하면서 두 패키지에 동일한 이름의 OrderController를 생성하였다.

    패키지는 서로 달랐지만 Spring은 Bean 이름을 클래스명 기준으로 생성하기 때문에 두 컨트롤러 모두 orderController라는 Bean 이름으로 등록되었다.

    그 결과 ConflictingBeanDefinitionException이 발생하며 서버가 실행되지 않았다.

    order 에 관한건 전부 중복 되었다.

    해결 방안 :

    지점용 컨트롤러의 역할이 명확하도록 클래스명을 변경하였다.

    역할을 클래스명에 포함하여 Bean 이름이 중복되지 않도록 수정하였다.

    결과 :

    Bean 이름 충돌이 해결되어 컨트롤러가 정상적으로 등록되었으며, 이후 동일한 구조의 클래스에서도 네이밍 규칙을 통일하게 되었다.
