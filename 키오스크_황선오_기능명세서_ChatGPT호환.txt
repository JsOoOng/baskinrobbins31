2. 키오스크 기능
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

──────────────────────────────────────────────────
2-1. 보유 쿠폰 목록 출력
──────────────────────────────────────────────────

1) 상세 흐름도
고객이 /point-discount에서 회원 조회 선택
→ 전화번호 입력
→ GET /api/users/points?phone={phone}
→ UserRepository가 전화번호에서 하이픈을 제거해 회원 조회
→ UserCouponRepository가 회원의 USER_COUPON 목록 조회
→ Coupon 정보와 사용 여부·만료일을 CouponDto로 변환
→ PointDiscountView.vue가 사용 가능한 쿠폰 목록 출력

2) 호출 API·URL·Method·역할·실제 화면 경로
- GET /api/users/points
- Query Parameter: phone
- 역할: 회원 포인트와 보유 쿠폰을 한 번에 조회
- 화면: /point-discount

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
- GET 요청이므로 Request Body 없음
- 사용 테이블: USERS, USER_COUPON, COUPON
- Repository: UserRepository, UserCouponRepository
- 응답 CouponDto:
  userCouponId, couponName, discountValue, discountType,
  isUsed, expiryDate

4) 백엔드 응답과 프론트 수신 방식
- UserPointResponseDto 반환
- pointBalance와 coupons 배열을 함께 반환
- 프론트는 response.data.coupons를 쿠폰 선택 목록에 반영

5) Vue 화면 이동
- /menu → /point-discount
- 쿠폰·포인트 선택 완료 → /order-confirm

6) 화면에서 동작하는 기능
- 전화번호로 회원 조회
- 쿠폰명, 할인 값, 할인 유형, 만료일 표시
- 사용한 쿠폰과 사용 가능한 쿠폰 구분
- 결제에 사용할 쿠폰 한 개 선택

7) 예외처리와 현재 구현 주의사항
- 등록되지 않은 전화번호는 현재 신규 회원을 만들고 포인트 0으로 응답한다.
- Repository 조회 결과에는 사용 완료 쿠폰도 포함될 수 있으므로 화면에서 isUsed를 확인해야 한다.
- 만료일이 지난 쿠폰을 서버 결제 단계에서도 다시 검증하는 로직이 필요하다.

8) 기능 핵심 정리
회원 전화번호를 기준으로 USER_COUPON과 COUPON을 조회해
고객이 결제에 사용할 수 있는 보유 쿠폰을 보여주는 기능이다.

──────────────────────────────────────────────────
2-2. 보유 쿠폰 사용
──────────────────────────────────────────────────

1) 상세 흐름도
고객이 보유 쿠폰 선택
→ PointDiscountView가 userCouponId를 basketStore.usedCouponId에 저장
→ OrderConfirmView가 결제 요청에 userCouponId 전달
→ POST /api/orders/{orderId}/pay
→ OrderService.processPayment()
→ UserCouponRepository로 회원 쿠폰 조회
→ AMOUNT 또는 PERCENT 방식으로 할인액 계산
→ 최종 결제액에서 쿠폰 할인 차감
→ UserCoupon.useCoupon()으로 사용 완료 처리
→ PAYMENTS.coupon_discount에 할인액 저장

2) 호출 API·URL·Method·역할·실제 화면 경로
- POST /api/orders/{orderId}/pay
- 역할: 쿠폰·포인트를 적용하고 결제 상세 저장
- 화면: /order-confirm

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
요청 예시:
{
  "paymentMethod": "CARD",
  "userCouponId": 15,
  "usePoints": false,
  "pointUsed": 0
}
- 사용 테이블: USER_COUPON, COUPON, ORDERS, PAYMENTS
- Repository/Mapper: UserCouponRepository, OrderMapper

4) 백엔드 응답과 프론트 수신 방식
- 성공 시 결제 완료 문자열 응답
- 프론트는 성공 후 장바구니를 비우고 /order-complete로 이동

5) Vue 화면 이동
- /point-discount에서 쿠폰 선택
- /order-confirm에서 결제
- 성공 시 /order-complete?orderId={orderId}

6) 화면에서 동작하는 기능
- 정액 할인(AMOUNT)
- 정률 할인(PERCENT)
- 선택 쿠폰 할인 반영
- 사용한 회원 쿠폰 재사용 방지용 상태 변경

7) 예외처리와 현재 구현 주의사항
- 쿠폰이 없으면 “쿠폰을 찾을 수 없습니다.” 예외가 발생한다.
- 쿠폰 할인액이 상품 금액보다 크면 사용을 차단한다.
- 현재 processPayment()에는 소유 회원 일치, isUsed, expiryDate의 최종 검증을 강화할 필요가 있다.

8) 기능 핵심 정리
화면에서 선택한 userCouponId를 결제 트랜잭션에 전달하고,
서버에서 할인액 계산과 쿠폰 사용 완료 처리를 수행하는 기능이다.

──────────────────────────────────────────────────
2-3. 보유 회원 포인트 출력
──────────────────────────────────────────────────

1) 상세 흐름도
고객이 전화번호 입력
→ GET /api/users/points
→ UserRepository.findByPhoneIgnoringHyphen()
→ USERS.point_balance 조회
→ UserPointResponseDto.pointBalance 반환
→ PointDiscountView에서 보유 포인트 출력

2) 호출 API·URL·Method·역할·실제 화면 경로
- GET /api/users/points?phone={phone}
- 역할: 회원 보유 포인트와 쿠폰 조회
- 화면: /point-discount

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
- Request Body 없음
- 사용 테이블: USERS
- Repository: UserRepository

4) 백엔드 응답과 프론트 수신 방식
- pointBalance가 null이면 0으로 변환
- 프론트가 보유 포인트와 최대 사용 가능 포인트를 화면에 표시

5) Vue 화면 이동
- /menu → /point-discount
- 포인트 확인 후 /order-confirm

6) 화면에서 동작하는 기능
- 전화번호 숫자 키패드 입력
- 회원 조회
- 보유 포인트 표시
- 전액 사용 가능 범위 안내

7) 예외처리와 현재 구현 주의사항
- 등록되지 않은 번호는 신규 USERS 레코드가 생성된다.
- 전화번호 형식과 중복 회원 정책을 서버에서 더 엄격히 검증할 수 있다.

8) 기능 핵심 정리
회원 전화번호로 USERS의 point_balance를 조회해
결제 전 고객에게 현재 사용 가능한 포인트를 보여주는 기능이다.

──────────────────────────────────────────────────
2-4. 보유 회원 포인트 사용
──────────────────────────────────────────────────

1) 상세 흐름도
고객이 사용할 포인트 입력
→ 보유 포인트와 주문 금액 범위 검증
→ basketStore.usedPoints에 저장
→ OrderConfirmView가 pointUsed 전달
→ OrderService.processPayment()
→ pointUsed를 남은 결제 금액 이하로 제한
→ USERS.point_balance에서 사용 포인트 차감
→ 최종 결제 금액의 5%를 포인트로 적립
→ PAYMENTS.point_used 저장

2) 호출 API·URL·Method·역할·실제 화면 경로
- POST /api/orders/{orderId}/pay
- POST /api/orders/toss/confirm
- 역할: 일반 결제 또는 Toss 승인 후 포인트 사용 처리
- 화면: /point-discount, /order-confirm, /toss/success

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
일반 결제 예시:
{
  "paymentMethod": "CASH",
  "userCouponId": 0,
  "pointUsed": 2000
}
Toss 승인 예시:
{
  "paymentKey": "...",
  "orderId": "kiosk_order_100",
  "amount": 7800,
  "pointUsed": 2000
}
- 사용 테이블: USERS, ORDERS, PAYMENTS
- Repository/Mapper: UserRepository, OrderMapper

4) 백엔드 응답과 프론트 수신 방식
- 결제 성공 문자열 반환
- 성공 후 결제 완료 화면으로 이동

5) Vue 화면 이동
- /point-discount → /order-confirm
- Toss 사용 시 /toss/success → /order-complete

6) 화면에서 동작하는 기능
- 포인트 직접 입력
- 최대 사용 버튼
- 보유 포인트 초과 방지
- 결제액 초과 사용 방지
- 결제 후 사용 포인트 차감 및 5% 적립

7) 예외처리와 현재 구현 주의사항
- 서버는 pointUsed를 finalAmount 이하로 제한한다.
- 현재 회원의 실제 보유 포인트보다 큰 값에 대한 서버 검증을 강화해야 한다.
- 포인트 차감과 적립은 결제 트랜잭션 안에서 함께 처리된다.

8) 기능 핵심 정리
프론트에서 선택한 포인트를 결제 요청에 포함하고,
서버가 최종 결제액 차감·회원 잔액 변경·적립까지 처리하는 기능이다.

──────────────────────────────────────────────────
2-5. 주문 최종 확인
──────────────────────────────────────────────────

1) 상세 흐름도
/order-confirm 진입
→ basketStore.cartItems 확인
→ 상품명·맛·옵션·수량 출력
→ 주문 유형·드라이아이스·회원 정보 유지
→ 현금·카드·간편결제 중 결제 방식 선택
→ POST /api/orders
→ OrderService가 재고 확인 후 주문과 상세 항목 저장
→ 일반 결제 또는 Toss 결제로 진행

2) 호출 API·URL·Method·역할·실제 화면 경로
- POST /api/orders: 주문 최종 생성
- POST /api/orders/{orderId}/pay: 일반 결제
- 화면: /payment, /order-confirm

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
{
  "orderType": "TOGO",
  "dryIceCount": 0,
  "dryIceMins": 0,
  "phoneNumber": "01012345678",
  "pointUsed": 1000,
  "userCouponId": 15,
  "kioskId": 1,
  "storeId": 1,
  "items": [
    {
      "productId": 1,
      "quantity": 1,
      "unitPrice": 9800,
      "options": [3],
      "flavors": [{"flavorId": 1, "quantity": 1}]
    }
  ]
}
- 사용 테이블:
  ORDERS, ORDER_ITEMS, ORDER_ITEM_OPTIONS, ORDER_ITEM_FLAVORS,
  PRODUCTS, ICECREAM_FLAVORS, PRODUCT_OPTIONS, KIOSKS, STORES
- Repository/Mapper:
  OrderRepository, OrderItemMapper, OrderItemOptionMapper,
  OrderItemFlavorMapper, OrderMapper

4) 백엔드 응답과 프론트 수신 방식
- 주문 생성 성공 시 orderId(Integer) 반환
- 프론트는 orderId를 결제 API와 완료 화면에 전달

5) Vue 화면 이동
- /point-discount → /order-confirm
- 이전 버튼 → /point-discount
- 결제 성공 → /order-complete

6) 화면에서 동작하는 기능
- 최종 상품·맛·옵션·수량 확인
- 결제 방식 선택
- 빈 장바구니 진입 차단
- 결제 실패 시 백업 장바구니 복원

7) 예외처리와 현재 구현 주의사항
- 장바구니가 비어 있으면 /kiosk로 이동한다.
- 주문 생성 직전에 재고를 다시 확인한다.
- 현재 주문 생성은 프론트가 전달한 unitPrice를 사용하므로 서버 가격 재조회 보강이 필요하다.

8) 기능 핵심 정리
장바구니와 할인 정보를 최종 검토하고,
주문 데이터를 DB 구조에 맞게 저장한 뒤 결제 단계로 연결하는 기능이다.

──────────────────────────────────────────────────
2-6. 장바구니 총금액에서 최종금액 계산
──────────────────────────────────────────────────

1) 상세 흐름도
장바구니 상품별 unitPrice × quantity 합산
→ 상품 자체 할인 금액 계산
→ 쿠폰 할인 계산
→ 쿠폰 적용 후 금액 확인
→ 포인트 사용액을 남은 금액 이하로 제한
→ 최종 결제 금액 계산
→ PAYMENTS의 금액별 컬럼에 저장

2) 호출 API·URL·Method·역할·실제 화면 경로
- GET /api/customer/basket: 장바구니 총액 조회
- POST /api/orders/{orderId}/pay: 서버 최종금액 계산
- 화면: /menu, /point-discount, /order-confirm

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
- 입력값: basketStore.totalPrice, usedCouponId, usedPoints
- 사용 테이블: ORDER_ITEMS, PRODUCTS, COUPON, USER_COUPON, PAYMENTS
- Repository/Mapper: OrderRepository, UserCouponRepository, OrderMapper

4) 백엔드 응답과 프론트 수신 방식
- 화면은 Pinia computed 값으로 예상 결제액 표시
- 서버는 결제 시 실제 할인 항목을 다시 계산해 PAYMENTS에 저장

5) Vue 화면 이동
- /menu에서 장바구니 총액
- /point-discount에서 할인 적용 예상 금액
- /order-confirm에서 최종 결제 예정 금액

6) 화면에서 동작하는 기능
- 상품 총액 합산
- 정액·정률 쿠폰 할인
- 포인트 할인
- 총 할인액과 최종 결제액 계산

7) 예외처리와 현재 구현 주의사항
- 서버 계산식:
  discountedAmount = order.getTotalPrice()
  couponDiscount = AMOUNT 또는 PERCENT
  finalAmount = discountedAmount - couponDiscount - pointUsed
  totalDiscount = productDiscount + couponDiscount + pointUsed
- Toss 요청 금액은 현재 화면에서 totalPrice - usedPoints로 계산되어
  선택 쿠폰 할인 반영 여부를 서버 계산과 반드시 일치시켜야 한다.
- 결제 금액의 최종 신뢰 기준은 서버가 되어야 한다.

8) 기능 핵심 정리
장바구니 총액에서 상품 할인, 쿠폰, 포인트를 순서대로 반영하여
실제 결제할 금액과 할인 내역을 산출하는 기능이다.

할인 종류별 상세 금액과 결제 수단·상태·일시를 PAYMENTS에 기록하는 기능이다.

3-4. 쿠폰 전체 유저에게 발급
──────────────────────────────────────────────────

1) 상세 흐름도
본사 관리자가 쿠폰 발급 모달 열기
→ 미발급 쿠폰 선택
→ POST /head/coupon/issue-all/{couponId}
→ INSERT INTO USER_COUPON ... SELECT FROM USERS
→ 이미 같은 쿠폰이 있는 회원은 NOT EXISTS로 제외
→ COUPON.is_issued_all = TRUE
→ 관리자 작업 로그 저장
→ 쿠폰 목록 재조회

2) 호출 API·URL·Method·역할·실제 화면 경로
- POST /head/coupon/issue-all/{couponId}
- 역할: 선택 쿠폰을 모든 회원에게 일괄 발급
- 화면: /head/coupons

3) 프론트 요청 JSON·사용 테이블·Mapper/Repository
- Request Body 없음
- PathVariable: couponId
- 사용 테이블: USERS, COUPON, USER_COUPON, ADMIN_ACTION_LOG
- Mapper:
  insertCouponToAllUsers(), updateCouponIssuedStatus()

4) 백엔드 응답과 프론트 수신 방식
- 성공: “쿠폰이 모든 유저에게 성공적으로 발급되었습니다.”
- 실패: 400과 실패 원인 문자열
- 성공 후 모달을 닫고 목록 재조회

5) Vue 화면 이동
- /head/coupons 내부 발급 모달에서 처리

6) 화면에서 동작하는 기능
- 전체 발급하지 않은 쿠폰만 선택
- 모든 USERS에 USER_COUPON 생성
- 중복 발급 방지
- durationDays 기준 expiry_date 계산
- 전체 발급 상태 O 표시

7) 예외처리와 현재 구현 주의사항
- 동일 회원·동일 쿠폰은 NOT EXISTS 조건으로 중복 생성하지 않는다.
- 발급 INSERT와 is_issued_all 변경은 하나의 서비스 트랜잭션으로 처리된다.
- 향후 가입 회원에게도 자동 발급할 것인지 별도 정책이 필요하다.

8) 기능 핵심 정리
쿠폰 마스터 한 건을 현재 전체 회원의 USER_COUPON으로 일괄 생성하고
발급 완료 상태와 관리자 이력을 함께 관리하는 기능이다.
