2. 기능을 넣은 배경과 관계 구조

기능 목록을 바로 보여주면 “왜 이 기능이 필요한지”가 약하게 보일 수 있습니다. 그래서 먼저 키오스크, 고객, 지점, 본사가 어떤 관계인지 설명하고 그 관계에서 필요한 기능을 도출하는 구조로 정리했습니다.

기능을 넣은 배경

주문 대기 감소고객이 직원에게 직접 말하지 않고 키오스크에서 메뉴, 옵션, 결제를 처리하면 매장 대기 시간을 줄일 수 있습니다.
주문 오류 감소콘/컵, 사이즈, 맛 개수, 포장 여부를 화면에서 직접 선택하게 하면 직원 전달 과정의 실수를 줄일 수 있습니다.
지점 운영 연결주문이 결제에서 끝나는 것이 아니라 지점 관리자 화면으로 넘어가야 제조, 픽업, 완료 상태를 관리할 수 있습니다.
본사 기준 통일메뉴명, 가격, 이벤트, 쿠폰, 판매기간을 본사에서 관리해야 전체 지점의 정보가 일관되게 유지됩니다.
보안과 추적결제, 관리자 행위, 개인정보 처리에는 권한 관리와 감사 로그가 필요합니다. 문제가 발생했을 때 누가 무엇을 했는지 추적할 수 있어야 합니다.
K · 키오스크고객이 매장 안에서 주문을 생성하는 입력 장치입니다. 메뉴 선택, 옵션 선택, 장바구니, 결제, 주문번호 발급을 담당합니다.
U · 사용자PC·모바일에서 회원가입, 로그인, 주문내역, 적립, 배달 주문, 문의 기능을 사용하는 고객 영역입니다.
B · 지점키오스크에서 들어온 주문을 실제로 제조하고 처리하는 매장 운영 영역입니다. 주문 상태 변경, 품절 처리, 재고 요청이 필요합니다.
H · 본사전체 지점이 공통으로 사용할 메뉴, 가격, 이벤트, 쿠폰, 지점 정보, 정책을 관리하는 기준정보 관리 영역입니다.
S · 시스템고객·지점·본사를 안전하게 연결하기 위한 공통 기반입니다. 인증, 권한, 로그, 결제 검증, 예외 처리를 담당합니다.

3. 플로우차트는 별도 HTML로 분리

키오스크 플로우차트 전용 화면

발표용 문서 안에 플로우차트를 억지로 넣으면 화면 크기 때문에 찌그러지거나 화살표가 어색하게 보일 수 있습니다. 그래서 플로우차트는 별도 HTML로 분리했습니다.

별도 파일에서는 고객 주문 흐름, 지점 처리 흐름, 본사·시스템 관리 흐름을 좌 → 우 방향으로 넓게 확인할 수 있습니다.

플로우차트 전용 HTML 열기

읽는 순서

고객 주문→ 옵션/장바구니→ 결제→ 지점 처리→ 본사/시스템 관리

플로우차트는 발표 중 “주문이 어디로 흘러가는지”를 설명할 때 사용하면 됩니다.

둘째, 장바구니와 결제를 거쳐 주문번호를 받습니다.

셋째, 지점 관리자는 실시간 주문을 확인하고 상태를 변경합니다.

넷째, 본점 관리자는 메뉴와 지점 정보를 통합 관리합니다.

이 흐름이 완성되면 1차 MVP는 발표 가능한 상태입니다.

7. 개발 기술 스택 (Tech Stack)

키오스크, 사용자 웹, 그리고 지점/본점 관리 인프라를 안정적으로 구동하고 유지보수하기 위한 표준 기술 스택 사양입니다.

Frontend

고객 인터페이스 및 비즈니스 대시보드

HTML5 / CSS3 JavaScript (ES6+) React / Vue.js

키오스크의 터치 반응성 및 관리자 대시보드의 실시간 화면 갱신을 고려하여 컴포넌트 기반 라이브러리를 채택합니다.

Backend

비즈니스 로직 및 통합 API 서버

Java 21 Spring Boot 5.x Spring Security MyBatis / JPA

대규모 트랜잭션 분산 처리와 보안 요건(암호화, 접근제어) 충족을 위해 기업 표준인 Spring 생태계를 구성합니다.

Data & Infra

안정성 기반의 데이터 관리 및 배포 인프라

MySQL / PostgreSQL Docker / AWS GitHub Actions

주문 정보의 정합성을 보장하는 RDBMS와, 실시간 주문 접수 및 장바구니 세션 처리를 위한 인메모리 캐시를 혼용합니다.

8. 브랜치 전략

기능 개발 브랜치에서 작업하고, dev에서 통합 테스트를 한 뒤, 검증된 내용만 main으로 배포합니다. 프로젝트에서는 브랜치명에 / 대신 _를 사용해 규칙을 통일합니다.

feature_기능명 fix_오류명 → dev 통합 → main 배포

기능 개발

새 기능은 기능 단위로 브랜치를 분리합니다.

feature_kiosk_order
feature_menu_option
feature_payment

오류 수정

버그 수정은 기능 개발과 분리해서 추적합니다.

fix_login_error
fix_payment_retry
fix_order_status

긴급 수정

운영 배포 후 치명 오류가 있으면 hotfix로 처리합니다.

hotfix_payment_error
hotfix_security_header

문서/정리

문서, 설정, 리팩토링은 목적별로 구분합니다.

docs_requirement_update
refactor_order_service
chore_ci_setting
항목 규칙 예시
기본 흐름 기능 브랜치에서 작업 → dev 병합 → 통합 테스트 → main 배포 feature_kiosk_order → dev → main
네이밍 feature/fix 방식은 많이 쓰지만, 이 프로젝트에서는 / 대신 _를 사용한다. feature_payment, fix_login_error
단위 하나의 브랜치는 하나의 기능 또는 하나의 오류 수정만 담당한다. feature_coupon 안에서 재고 기능까지 같이 개발하지 않음
dev 여러 개발자의 기능을 모아 테스트하는 통합 브랜치로 사용한다. 키오스크 주문 + 지점 주문관리 연동 테스트
main 발표/배포 가능한 안정 버전만 병합한다. 직접 커밋하지 않는 것을 원칙으로 한다. 발표 전 최종 안정 버전
hotfix main 배포 후 긴급 오류는 hotfix에서 수정하고 main과 dev 양쪽에 반영한다. hotfix_payment_error → main, 이후 dev에도 반영
커밋 메시지 작업 유형이 보이도록 짧게 작성한다. feat: 키오스크 장바구니 추가, fix: 로그인 오류 수정
병합 기준 기능 확인, 충돌 해결, 기본 실행 테스트 후 병합한다. 주문 생성 API 호출 성공, 화면 이동 확인

9. 발표 결론

이 시스템은 고객의 주문·결제 흐름을 중심으로, 지점 관리자의 주문 처리와 본점 관리자의 기준정보 관리를 연결합니다.

1차는 주문 가능 상태, 2차는 실사용 편의성, 3차는 배달·물류·운영 자동화로 확장합니다.
